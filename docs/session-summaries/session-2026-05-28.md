# Session Summary — 2026-05-28

## Session Goal

Add rate limiting to the RoomMatch FastAPI backend to protect public auth endpoints from brute-force attacks and prevent abuse of authenticated user routes, with Sentry observability for production monitoring.

---

## What Was Completed

All six planned tasks were completed this session.

**Task 1 — Install slowapi and configure Redis/in-memory limiter**
Added `slowapi==0.1.9` to `backend/requirements.txt`. Created `backend/app/limiter.py` with a shared `Limiter` instance. The storage URI is built at startup: if `UPSTASH_REDIS_URL` is present (gated behind `UPSTASH_REDIS_REST_URL`/`_TOKEN` also being set), it uses Redis; otherwise falls back to `memory://` for local development. The key function (`_get_user_or_ip`) keys by the first 32 characters of the Bearer token when present, or by IP address for unauthenticated requests.

**Task 2 — Apply rate limits to auth routes**
Decorated auth route handlers in `backend/app/routers/authRoutes.py` with `@limiter.limit(...)`:
- `POST /api/auth/register` → 3 / hour
- `POST /api/auth/login` → 5 / 15 minutes

Added `request: Request` as the first parameter to each handler (required by slowapi). Wired `SlowAPIMiddleware` and the rate-limit exception handler into `main.py`.

**Task 3 — Apply rate limits to user routes**
Decorated all handlers in `backend/app/routers/userRoutes.py`:
- `POST /api/users/{id}/like` → 100 / hour
- `POST /api/users/{id}/chat/{partner_id}` → 30 / minute
- `POST /api/users/{id}/upload-photo` → 10 / hour
- All other user/admin routes → 60 / minute (default)

**Task 4 — Log rate-limit hits to Sentry**
Replaced the default slowapi 429 handler with a custom `_rate_limit_handler` in `main.py`. When `SENTRY_DSN` is set, each 429 is captured via `sentry_sdk.capture_message` at `warning` level with `route`, `ip`, and `user_id` as extras. The `user_id` is extracted by a `_user_id_from_request` helper that decodes the Bearer token directly via `decode_token` (from `app.auth.utils`) and reads the integer `sub` claim — no DB call. This is necessary because FastAPI `Depends`-based auth does not run when the rate limit middleware fires before the route handler. Sentry is initialized conditionally on startup (`traces_sample_rate=0.0` to avoid performance overhead). When `SENTRY_DSN` is absent (local dev), the handler still returns a well-formed 429 JSON response with a `Retry-After: 60` header.

**Task 5 — Write pytest tests for each rate limit**
Created `backend/test_ratelimits.py` with a self-contained FastAPI test app (in-memory limiter, fixed `X-Forwarded-For` IP for deterministic keying). Six tests covering:
- Login: 5 successful → 6th is 429
- Register: 3 successful → 4th is 429
- Like: 100 successful → 101st is 429
- Chat send: 30 successful → 31st is 429
- Photo upload: 10 successful → 11th is 429
- Default limit: 60 successful → 61st is 429

All tests assert both the 429 status code and the presence of a `Retry-After` header.

**Task 6 — Document rate limiting setup**
Created `docs/rate-limiting.md` covering slowapi overview, all env vars, the full rate limit table, Upstash Redis provisioning steps, curl/PowerShell commands to trigger each 429 locally, and Sentry monitoring guidance. Updated `README.md` to add Rate Limiting to Features, slowapi/Upstash/Sentry to the Tech Stack table, three new env vars to the Environment Variables table, and a link to the doc in Security Notes.

## IDOR Security Sprint

A second sprint ran in the same session to address Insecure Direct Object Reference (IDOR) vulnerabilities across the backend.

### What Was Found

**3 Critical** — `chatRoutes.py` had no authentication at all. Any unauthenticated HTTP client could send messages as any user, read any chat thread, or list any user's conversations.

**15 High** — All user-scoped routes in `userRoutes.py` were authenticated (valid JWT required) but did not verify the caller's identity against the `{user_id}` path parameter. Any valid token holder could read or mutate any other user's data: profile, likes, matches, notifications, chat, and photo.

### What Was Fixed

**Task #7 (backend)** — Added two new dependencies to `backend/app/auth/dependencies.py`:
- `get_current_user_or_403`: resolves the authenticated user, then asserts `current_user["id"] == user_id` from the path param — returns 403 otherwise.
- `verify_match_exists`: queries `matches_collection` for a confirmed match between `user_id` and `partner_id` — returns 403 if none exists.

**Task #8 (backend)** — Replaced `Depends(get_current_user)` with `Depends(get_current_user_or_403)` on all user-scoped routes in `userRoutes.py`. Added `Depends(verify_match_exists)` to the two chat endpoints that carry `partner_id` as a path param.

**Task #9 (auth)** — Applied `get_current_user_or_403` to all three `chatRoutes.py` endpoints. Confirmed `authRoutes.py` and `matchingRoutes.py` have no IDOR surface.

**Task #10 (tests)** — Wrote negative pytest tests confirming 403 is returned when a caller uses another user's ID in the path.

**Task #11 (docs)** — Wrote `backend/SECURITY_AUDIT_IDOR.md` with a full route-by-route audit table, executive summary, vulnerability count by severity, and remediation status.

**Task #12 (backend)** — `POST /api/chat/{user_id}/send` could not use `verify_match_exists` because the receiver (`receiverId`) is in the request body, not a path parameter. Resolved with an inline `matches_collection.find_one` query directly in the handler, raising 403 if no confirmed match exists between `user_id` and `request.receiverId`.

### What Remains

Nothing — all 18 previously vulnerable routes are fully remediated. No open IDOR items.
